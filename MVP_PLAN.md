## 1. Project Summary

This project is a local-first desktop application for managing a hospital contact directory. The primary users are administrative staff working in a public hospital environment. The application must optimize for speed of search, clarity of information, offline operation, maintainability, and safe handling of sensitive phone numbers.

The current source of truth is an `.ods` spreadsheet that mixes people, services, departments, rooms, supervision contacts, centers, and other operational entries. The MVP must not replicate the spreadsheet structure directly. Instead, it must normalize the data into a cleaner JSON-based model.

The application will:

- run locally without internet access
- have no server dependency
- persist data in local JSON files
- support manual JSON export and sharing between coworkers
- support JSON import
- support CSV import as the migration bridge from the current `.ods`
- support manual backup plus automatic backup before import
- be responsive
- include tests
- keep all generated documentation in English
- keep the application UI in Spanish for now

## 2. Confirmed Product Decisions

### 2.1 Platform and Technical Direction

- Platform: `React + Electron`
- Build base: `React + TypeScript + Vite`
- Import formats in MVP: `JSON + CSV`
- Real migration path: `ODS -> CSV -> application`
- Import strategy: full replace
- Backup strategy: manual backup plus automatic backup before every import

### 2.2 Data Modeling

- Main model strategy: one base record with type variants
- Record identity: internal autogenerated `id` plus optional `externalId`
- Required fields: `displayName + type + at least one contact method or location`
- Classification: `type + primary area + tags`
- Catalog strategy: controlled catalogs for `type` and `area`, mixed/free text for the rest
- Search support: optional aliases for abbreviations, legacy names, and operational shorthand
- Phone strategy: structured list of phones
- Extension strategy: extension is a dedicated field within each phone item
- Location strategy: minimal structured location with optional building, floor, room, and free text
- Audit strategy: `createdAt`, `updatedAt`, `createdBy`, `updatedBy`
- Inactive records: hidden by default, visible through filters

### 2.3 UX and Interaction

- Search engine: `Fuse.js`
- Edit mode: dedicated form page
- Main navigation: primary workspace plus dedicated screens for edit, settings, and import/export
- Search result density: name, type, unit, main phone, and privacy flags
- Sensitive phones: always visible, but with strong permanent visual warnings
- CSV import flow: preview and summary before confirmation

### 2.4 Frontend and Validation

- State management: `Zustand`
- Styling: `Tailwind CSS`
- Data validation: `Zod`
- Tests: `Vitest`, `Testing Library`, and `Playwright` for critical flows

## 3. Recommended MVP Architecture

### 3.1 Layered Architecture

The recommended architecture has three layers:

1. `Electron main process`
   - filesystem access
   - import/export operations
   - backup creation
   - JSON persistence
   - CSV parsing and normalization

2. `Electron preload`
   - secure, typed bridge between Electron main and React renderer

3. `React renderer`
   - search and filters
   - record detail
   - forms
   - navigation
   - UI state

This separation keeps filesystem logic out of the UI and provides a maintainable base for future growth without introducing a backend.

## 6. Data Flow

### 6.1 Startup Flow

1. the app starts
2. Electron main resolves the default local data paths
3. if the data files do not exist, the app creates them from the bundled initial templates
4. if the settings file exists, the app loads user overrides
5. the main process loads and validates the JSON dataset
6. if the dataset file is invalid or corrupted, the app shows a recovery state and blocks normal editing until the user imports a valid JSON backup or resets to a clean dataset
4. preload exposes a typed API
5. the renderer hydrates the store
6. the search index is built

### 6.2 Edit Flow

1. the user opens the create or edit form
2. the form validates data with `Zod`
3. the renderer sends a save request through preload
4. the main process persists the JSON file
5. the renderer reloads data and refreshes the search index

### 6.3 Import Flow

1. the user selects a CSV or JSON file
2. the main process parses and normalizes the file
3. validation runs
4. the renderer receives a preview summary
5. the user confirms the import
6. the main process creates a backup
7. the dataset is replaced
8. the renderer reloads the dataset

## 7. State Strategy

The application state should be divided into lightweight logical areas:

- `dataset state`
  - loaded records
  - dataset metadata
- `search state`
  - query
  - filters
  - selected result
- `ui state`
  - loading flags
  - dialogs
  - toasts
  - panel visibility
- `settings state`
  - editor name
  - data and backup paths
- `import state`
  - preview
  - warnings
  - errors

The JSON files remain the source of truth. Zustand only manages runtime application state.

## 8. Persistence Strategy

The recommended persistence structure is:

```txt
<app-data-directory>/
  data/
    contacts.json
    settings.json
  backups/
    contacts-2026-04-13T12-00-00.json
```

The default storage root for the MVP should be the application data directory provided by Electron for the current user profile. The app should own this location by default and should not require manual path selection on first launch.

Recommended first-run behavior:

- create the app data directory if it does not exist
- create `data/contacts.json` from a bundled sample or empty template
- create `data/settings.json` with default values
- create `backups/` only when needed or during setup
- open the main workspace immediately after initialization

Path editing remains available in Settings, but the default experience should work without any manual filesystem configuration.

### 8.2 `settings.json`

## 10. Validation Strategy

Validation should exist at three levels:

### 10.1 Schema Validation

Use `Zod` for:

- dataset files
- records
- phones
- emails
- location
- settings

### 10.2 Business Validation

Rules include:

- every record must have `displayName`
- every record must have `type`
- every record must have at least one contact method or a location
- phone privacy flags must be explicit and valid

### 10.3 Import Validation

- invalid rows must be reported
- imports must show a preview before confirmation
- the app must explain failures in plain operational language

## 11. Backup Strategy

The MVP backup strategy should be intentionally simple:

- manual backup from the import/export screen
- automatic backup before every confirmed import
- timestamped backup files
- no automatic cleanup in the MVP
- restore through normal JSON import instead of a dedicated restore subsystem

## 20. Sensitive Phone Privacy Behavior

Sensitive phones are not hidden in the MVP. The safety strategy is based on high-clarity presentation.

### 20.1 Rules

- `confidential = true`
  - permanent high-visibility badge
- `noPatientSharing = true`
  - stronger warning badge with explicit text
- if both are true
  - both warnings are displayed
- in result cards
  - if a non-sensitive primary phone exists, it should be preferred as the visible result-card phone
  - if only sensitive phones exist, the phone may still be visible
  - when a sensitive phone is shown in result cards, it must include explicit inline risk text, not only a badge
- in detail view
  - warning block appears above the phone list
- in forms
  - privacy controls must be explicit and readable

Recommended result-card labels for sensitive numbers:

- `Internal confidential`
- `Do not share with patients`
- `Sensitive internal number`

## 27. Main Risks and Scope Control

### 27.1 Highest Risks

- source CSV quality after ODS conversion
- overcomplicated import mapping
- UI density in the main workspace
- underestimating Electron packaging and file path handling

### 27.2 What Must Not Expand During the MVP

Do not add the following during the four-week MVP:

- direct ODS support in the application
- dataset merge logic
- advanced backup restore workflows
- advanced auditing
- authentication
- roles and permissions
- online sync
- analytics dashboards
- theming systems

### 27.3 Scope Guardrail

If a feature does not improve one of these six MVP outcomes, it should be deferred:

- faster search
- clearer detail view
- safer handling of sensitive numbers
- easier create and edit flow
- reliable import and export
- safer local data persistence

## 28. Definition of MVP Success

At the end of the four-week cycle, the MVP is successful if a hospital administrative user can:

- load contacts from a local JSON file
- import normalized data from CSV
- search quickly across multiple fields
- filter by record type and area
- view clear record details
- create and edit records
- identify sensitive phone numbers safely
- export JSON for sharing
- create backups
- restore by importing a previous JSON backup
